# Repository Guidelines

## Project Structure & Module Organization
`site.yml` orchestrates every host group and its roles. Role logic stays under `roles/<role>/` (tasks, templates, handlers), with shared defaults in `group_vars/` and host-specific overrides in `host_vars/`. `ansible.cfg` binds the repo to `hosts.ini` and `vault-client.sh` (which dispatches on vault-id; see *Vault ids* below), so running from the root picks up the correct inventory and vault passwords automatically. Supporting folders: `packer/` builds the QEMU images, `terraform/` keeps Cloudflare DNS code and state, `wireguard/` stores VPN peers, `test/` holds the harness, inventories, and logs, and `notes/` carries long-form design notes (plans, migration journals, rationale) referenced from code comments and CLAUDE.md pointers — the canonical landing zone for content that doesn't belong inline.

## Build, Test, and Development Commands
- Bootstrap tools: install [mise](https://mise.jdx.dev), `mise trust`, then `mise install` from the repo root — pins terraform, packer, python, uv, and shellcheck. `python.uv_venv_auto` auto-creates and sources `.venv`, so `uv sync` populates Python deps (ansible, ansible-lint, black, yamllint) on first directory entry. 1Password CLI must be signed in for the `op://` env vars in `mise.toml` to resolve.
- Configure everything: `ansible-playbook site.yml --limit prod` (`ansible.cfg` already points at `hosts.ini`; set `--tags` to narrow scope).
- Focus on one service or host: `ansible-playbook wireguard.yml -l lab --tags wireguard`.
- Manage DNS: `mise run tf init`, `mise run tf plan`, then `mise run tf apply` — the `tf` task `cd`s into `terraform/` and forwards everything to `tofu` (use `--` for flags mise might intercept, e.g. `mise run tf plan -- -refresh=false`).
- Refresh the integration image when the base OS changes: `mise run packer:build` builds every source defined in `packer/qemu.pkr.hcl` in parallel (packer's multi-source orchestration handles the fan-out). Pass one or more source names explicitly to narrow it: `mise run packer:build box` or `mise run packer:build pug lab`. `--ubuntu noble` targets a different release. Three sources: `box` (minimal ZFS-on-root, single-disk rpool, the only image push CI consumes), `pug` (matches the pug prod host: single rpool + apoc), `lab` (matches the lab prod host: mirror rpool + dozer/tank/mouse). `pug` and `lab` exist to exercise the `chroot.sh` / `provision.sh` / `pools.sh` scripts against real prod shapes and to exercise producer roles against the real multi-pool layout (box runs them too, but with `zfs_dozer_filesystem` / `zfs_tank_filesystem` repointed at rpool so the datasets land flat under rpool/); the harness reaches them via `--machine pug` / `--machine lab` on demand. See "Test Environment Design" below.
- Lint everything: `mise run lint` runs ansible-lint, terraform/packer fmt -check, black --check, yamllint, and shellcheck in parallel; `mise run fmt` applies fixes (ansible-lint --fix, terraform/packer fmt, black, and `mise fmt` for `mise.toml` itself).
- Inner-loop ansible-lint: prefer `mise run lint:ansible-changed` (~4s; lints YAML changed vs `origin/master` + uncommitted + untracked) over the full `mise run lint:ansible` (~40s, dominated by ~100 unskippable per-role `ansible-playbook --syntax-check` subprocesses). Override the base ref with `LINT_BASE=<ref>`. Run full `mise run lint` before pushing.

## Coding Style & Naming Conventions
Use two-space YAML indentation and descriptive `name` values. Namespace variables per role (e.g., `samba_share_path`). Shell helpers in `packer/scripts/` should be Bash with `set -euo pipefail`. Before committing, run `mise run lint` (or `mise run fmt` to autofix the formatters) to avoid churn.

**Underscores, not hyphens** in identifiers we author: role names, file names under `roles/`, systemd unit names, vars, directory names under `/mnt/services/<svc>/`, etc. — `clickhouse_logs` not `clickhouse-logs`, `service_user.yml` not `service-user.yml`. Editor double-click on `foo_bar` selects the whole token; on `foo-bar` it stops at the hyphen, which makes search/replace and code-aware navigation worse. The exceptions are names dictated by upstream (image tags, package names, K8s-flavored YAML keys, etc.) — those keep their hyphens.

### Role Conventions
- Centralize downloadable artifact metadata in `roles/<role>/vars/main.yml`. Keep the full URL **adjacent to its sha256** keyed by `ansible_architecture` (`x86_64` / `aarch64`), so a human auditing a checksum can see exactly which URL it pins without recomposing it from a version constant.
- For tasks that mutate state but should run only once (downloads, registrations, token fetches), gate them with `args: creates: <sentinel>` rather than `changed_when: false`. Skipped tasks don't count against the harness's idempotence check, and `changed_when: false` lies about what the task actually does.
- Test-only branches read the harness boolean `qemu_test`, set directly in the test fixture host_vars: `host_vars/box.yml` and `host_vars/minimal.yml` (box and minimal are test-only inventory hosts), plus `host_vars/lab-qemu.yml` / `host_vars/pug-qemu.yml` (overlays applied by the harness on top of inventory-loaded `host_vars/{lab,pug}.yml` when `--machine lab/pug` is used). Use `qemu_test` in `when:` / `ternary` rather than per-machine name comparisons. The `minimal` variant additionally sets `qemu_test_minimal: true` in its host_vars for the small-VM-vs-full-VM distinction (currently only `roles/zfsbootmenu` reads it).
- Per-role test hooks live alongside the role's tasks:
  - `tasks/_setup.yml` — pre-role fixture bringup (e.g. start a local gitea instance for the role to register against).
  - `tasks/_verify.yml` — post-converge assertions exercising the role's real behavior: hit the service over its port, trigger the failure path, exercise the safety harness. **Only ever invoked by `test/testrole.py` / `test/testall.py` against a qemu VM** — `site.yml` and `mise run ansible --tags <role>` against prod never run it. That contract is what lets _verify scaffolding (netns/veth fixtures, sentinel tables, test-only listeners) sit alongside the role's real tasks without an explicit `when: qemu_test` gate; if you ever need _verify to run on prod, gate the destructive bits first. Stat-only ("file present", "unit active") checks drift toward tautology — they pass on builds where the machinery is wired but the function is broken (e.g. a rollback harness whose snapshot is captured after the destructive change). Fall back to state-only assertions when functional testing is genuinely impossible (e.g. requires a chaos event the harness can't safely produce), and say why in a comment.
  - `tasks/_test.yml` — legacy alias for `_setup.yml`, still supported; many roles still use it.
- Tasks whose target depends on a freshly-created user, group, or directory must be gated with `when: not (ansible_check_mode and <svc>_user.changed)` (or whatever resource was just created). In check mode the prerequisite task only *reports* it created the user/dir — nothing is actually written — so a downstream `file:` / `copy:` / `template:` will fail with "no such user" or "no such directory". The gate skips the dependent task only when both conditions hold (we're in check mode AND the upstream resource was reported as changed); on a real run the resource really does exist, so the dependent task proceeds. This is a load-bearing idiom — every service role that creates a system user before writing files relies on it.
- Use bare module names (`command:`, `apt:`, `deb822_repository:`, `assert:`) inside a role's task files; don't mix FQCN (`ansible.builtin.command:`) with bare in the same file. Drift between the two reads as "new code is FQCN, old isn't" and encodes a convention that doesn't exist. Pick bare so the form matches the older `import_role`/`include_tasks` calls scattered across the repo.
- Every `copy:` / `template:` writing a config file should carry a best-effort `validate:` that *parses* (not just reads) the rendered file. The point is to catch jinja quoting bugs and vault-substitution typos before the broken file lands on disk — many services accept-then-overwrite-with-defaults on parse failure, silently losing config. Pick the parser matching the format: `python3 -c 'import yaml,sys; yaml.safe_load(open(sys.argv[1]))' "%s"` for YAML, `python3 -c 'import configparser; configparser.ConfigParser(allow_no_value=True).read("%s")'` for INI (see [roles/nut_server/tasks/main.yml](roles/nut_server/tasks/main.yml)), `nginx -t -c %s` for nginx, `visudo -cf %s` for sudoers. Best-effort: if no syntax-checker exists for the format, omit `validate:` rather than invent a fake one — `validate: true` is worse than nothing because it lies about what's being checked.
- `/mnt/services/<svc>/` is for service *state* (configs, databases, secrets, persistent data — anything that must ride ZFS snapshots/replication). Service *code* shipped from the repo (`roles/*/files/*.py`, scripts deployed via `copy:`) belongs at `/opt/<svc>/` — it's reproducible from the role and doesn't need to be snapshotted. Putting code under `/mnt/services/` would inflate snapshot diffs on every edit and conflate "data I can't lose" with "code I can re-apply." [roles/homepage/tasks/main.yml](roles/homepage/tasks/main.yml)'s split is the canonical example: YAML configs land under `/mnt/services/homepage/`, the Python sidecar binary lands under `/opt/homepage_alerts/`.

### Service Ports
Ports each host-level service publishes live in `group_vars/all.yml` under `service_ports:` — single source of truth for unit templates, nginx `proxy_pass`, firewall rules, and cross-role consumers (e.g. pihole's dnscrypt upstream). Covers both loopback-published podman services (`--publish 127.0.0.1:{{ service_ports.<svc> }}:<container_port>/tcp` + `nginx_site_args.proxy_pass: http://localhost:{{ service_ports.<svc> }}/`) and native daemons whose port is referenced by the firewall or another role (e.g. `dnscrypt_proxy` consumed by `roles/firewall` and `roles/pihole`). Migrate roles to consume from there as you touch them; until a role is migrated its port literal lives in multiple places. When allocating a new port, grep `service_ports:` first.

### ZFS Site Mountpoints
Per-site dataset gates live in `group_vars/all.yml` under `zfs_has_<name>_mount:` (services / scratch / media / data / brumath / eckwersheim / minio). Producer/consumer split: **producers** (`roles/data`, `roles/media`, `roles/scratch`, `roles/minio`, `roles/services`, `roles/libvirt`) create the dataset unconditionally on every `zfs_root` host they run on and **never read the flag** — the dataset is cheap, ZFS-managed, and has no consumer side-effect by itself. **Consumers** (service unit templates and samba's mount overrides) gate on `zfs_has_<name>_mount` to decide whether to bind-mount `/mnt/<name>` into their service. Test fixtures (`host_vars/box.yml` for the default push-CI variant, plus `host_vars/{lab,pug}-qemu.yml` overlays for on-demand `--machine lab/pug` runs) flip every flag to `false`, so the qemu VMs exercise consumer roles cleanly without prod-data dependencies while producer roles still get exercised on the test VM. Adding a new site mount: register the flag in `group_vars/all.yml` (defaulting `true`), add a producer task (extend `roles/data` for personal-data sites, or a sibling producer), and gate `BindReadOnlyPaths=/mnt/<name>` in every consuming `.service.j2`. Don't gate the producer — it would silently turn the role into a no-op under the default test machine.

### Homepage Bookmarks
Any new user-facing service (anything with an nginx subdomain) gets a bookmark in [roles/homepage/templates/bookmarks.yaml.j2](roles/homepage/templates/bookmarks.yaml.j2) — pick the right section (Media / Infra / Personal), follow the existing `abbr: XX` + `icon: sh-<name>.png` + `href: https://<subdomain>.{{ inventory_hostname }}.{{ domain }}/` shape. Icons resolve through selfh.st (the `sh-` prefix); if the service has no selfh.st icon, omit the `icon:` line and the 2-letter `abbr:` becomes the visible fallback. Adding bookmarks doesn't need a separate test run — the YAML structure is enforced by example.

### Container Healthchecks
Every podman `*.service.j2` unit must declare `--health-cmd` (and `--health-startup-cmd`) so systemd can distinguish "process running" from "service serving". The check executes *inside* the container, so it has to invoke something the image actually contains. In order of preference:

1. **`curl` / `wget` already in the image** — most application images ship one; grep the image's Dockerfile before adding a sidecar.
2. **Python `urllib.request` with the URL in argv** — python-based images (e.g. `linuxserver/profilarr`). Pass the URL as `argv[1]` rather than embedded in the Python literal, and use the JSON-array form so podman gets exec semantics: `--health-cmd '["python","-c","import urllib.request as u, sys; u.urlopen(sys.argv[1], timeout=1)","http://localhost:PORT/"]'`. The naïve `python -c 'u.urlopen("http://...")'` shape *fails* in a systemd unit — the `\"` escapes around the URL get unescaped during the systemd→podman handoff, podman can't JSON-parse the result, falls back to `/bin/sh -c '[python,-c,...]'`, and the healthcheck logs `not found`. Anything python where the script needs internal string literals belongs in argv too, for the same reason.
3. **Service-native CLI** — `redis-cli ping`, `mosquitto_sub`, `dig +short @127.0.0.1`, the app's own binary if it exposes a probe subcommand. Strongest signal when available.
4. **Bind-mount static curl via the `static_curl` role** — escape hatch for distroless images (`gcr.io/distroless/*`) that have neither shell nor HTTP client. `static_curl` installs `/usr/local/bin/curl_static` (sha256-pinned, statically-linked, from the `stunnel/static-curl` releases). Add the role to `site.yml` (it lives under *Base installation*) and bind-mount `/usr/local/bin/curl_static:/curl_static:ro` into the unit. Because podman wraps a string `--health-cmd` in `/bin/sh -c` — which a distroless image doesn't have — use the **JSON-array form** to get exec semantics: `--health-cmd '["/curl_static","--fail","--silent","-o","/dev/null","http://localhost:PORT/healthz"]'`.

Do not drop the healthcheck and lean on external monitoring (kuma, the `_verify.yml` task) instead — without an in-container check, `--sdnotify=healthy` can't gate the unit's `active` state, podman won't auto-restart on HTTP failure, and the unit reports liveness only at the OS-process level. If none of the four mechanisms above is feasible for an image, treat that as a blocker, not a license to skip.

### Container Secrets

Three paths exist for getting a podman secret into a containerized app. Preferred order:

1. **App-native `*_FILE` setting.** Pair `--secret=<name>,type=mount,target=<basename>` with `--env XXX_FILE=/run/secrets/<basename>`. The app reads the file directly; the secret never lands in any env var at runtime. Django supports `EMAIL_HOST_PASSWORD_FILE` natively; paperless-ngx supports `_FILE` on every `PAPERLESS_*` env. Canonical example: `EMAIL_HOST_PASSWORD_FILE` in [roles/healthchecks/templates/healthchecks.service.j2](roles/healthchecks/templates/healthchecks.service.j2).
2. **linuxserver `FILE__<VARNAME>` prefix.** Pair `--secret=<name>,type=mount,target=<basename>` with `--env FILE__VAR=/run/secrets/<basename>`. Every `linuxserver/*` image's s6-overlay init reads the file at startup and exports the env var to its s6 child, so the secret value isn't visible in the unit file or in `podman inspect`. Use when the app expects an env var and lacks its own `*_FILE` support. Canonical example: `SUPERUSER_PASSWORD` / `SECRET_KEY` in [roles/healthchecks/templates/healthchecks.service.j2](roles/healthchecks/templates/healthchecks.service.j2). linuxserver-specific — won't apply to non-linuxserver images.
3. **`--secret=<name>,type=env,target=VAR`.** Last-resort: the secret lands in the container's environment, visible to `podman inspect` and `/proc/<pid>/environ`. Acceptable on single-operator hosts where root is the only attacker that matters, but prefer 1 or 2 when the consumer supports it. Currently used by `openobserve` (`ZO_ROOT_USER_PASSWORD`), `openproject` (`SECRET_KEY_BASE`), and `paperless` (`PAPERLESS_SECRET_KEY`) — all non-linuxserver. paperless-ngx supports `_FILE` natively; the other two need upstream-doc verification before migrating.

### Container User Namespacing

Every podman unit uses the **fake-root** uidmap pattern: `--user 0:0` plus `--uidmap=0:0:65536 --uidmap=+0:{{ <svc>_user.uid }}:1` (and the matching gidmap). In-container root maps to the dedicated host system user created by `service_user`; everything else falls through 1:1 from the base mapping. The `+0:N:1` override is what makes a namespace-escape land at an unprivileged host uid rather than real root.

That default is sufficient when the container runs entirely as in-container root (healthchecks with `PUID=0`, etc.). It is **not sufficient** when the image's entrypoint drops privileges mid-startup to a baked-in non-root user — e.g. ClickHouse → in-container uid 101 `clickhouse`, MongoDB → uid 100 `mongodb`. With the bare 1:1 fall-through, those uids land on whatever the host happens to have at the same number (commonly `systemd-resolve` / `systemd-journal`), which is wrong for two reasons: `ls -la` lies about who owns the data, and ZFS replication breaks portability because the destination host's package layout may put a *different* system user at that uid — silently changing apparent ownership across a `zfs send | recv`.

For every privilege-drop the image **actually performs** (not might-someday-perform — confirm by reading the entrypoint, or by checking host-side file ownership after a test run), allocate a dedicated host system user (`<svc>_<role>`, e.g. `hyperdx_clickhouse`) via `service_user` with `config_dir: false`, and add a `+N:<uid>:1` override (and matching gid override) where `N` is the in-container uid the daemon runs as. Also add an ownership assertion in `_verify.yml` — `stat:` the bind-mount and `failed_when:` the owner is wrong — so a regression in the mapping is loud rather than silent. See [roles/hyperdx/](roles/hyperdx/) for the canonical example: ClickHouse drops to uid 101 so it gets the mapping; MongoDB has a baked-in uid 100 user but the image's entrypoint doesn't switch to it, so no `+100` is needed (add it the day the image starts using it, not before).

Don't conflate "the container runs as root" with "the daemons inside the container run as root" — many images intentionally split the two. Skipping the per-daemon mapping only matters cosmetically until you start replicating data across hosts or grepping `ls -la` during an incident, at which point it becomes a real diagnostic hazard.

### Prefer system-scope systemd units

Default for new timers / services is **system-scope** (`/etc/systemd/system/`, running under `systemd[1]`). Reach for user-scope (`systemctl --user`, `user@<uid>.service`, linger) only when the work fundamentally requires it — today that's just rootless podman, which needs a per-user `podman.socket` on the user bus ([roles/gitea_runner/](roles/gitea_runner/), [roles/github_runner/](roles/github_runner/)).

Reasons:
- **One operational vocabulary.** `systemctl status foo`, `systemctl restart foo`, `journalctl -u foo` work without `--user --machine=<u>@.host` gymnastics.
- **Monitoring works out of the box.** netdata's `systemdunits` go.d collector attaches to the system bus only; user-scope failures don't fire `systemd_service_unit_failed_state`. The `systemd_timers` chart.sh's `systemctl show LastTriggerUSec` also queries the system bus, so a user-scope timer is "overdue" forever in the chart. Going system-scope closes both gaps.
- **Single mental model for permissions.** The system unit's `User=` / `Group=` directives are the only privilege-drop axis. No second model for the user manager's lifecycle (linger, `user@<uid>.service`, per-user subuid maps for the manager process itself).

When the work needs to run as a non-root identity but stay system-scope, use `systemd_timer`'s `run_as:` arg (emits `User=<name>` / `Group=<name>` in the unit). [roles/getmail/tasks/main.yml](roles/getmail/tasks/main.yml) is the canonical example — runs as the operator so the maildir is directly browsable, but the unit itself is system-scope.

### Helper Roles
Several helper roles factor out duplicated patterns. Prefer them over re-implementing the boilerplate; a hand-rolled implementation in a service role is a migration candidate.

- `service_user` — `import_role: name: service_user, tasks_from: user` with `service_user_args: { name: <svc> }` creates the standard system group + nologin user (`/nonexistent` home) and `/mnt/services/<svc>` config dir, and exposes `<svc>_user.uid` / `.group` to the caller. Optional dict keys: `extra_groups` (list, e.g. `[kvm]`), `createhome: true`, `home`, `shell`, `config_dir: false` (opt out of `/mnt/services/<svc>` for stateless services). Inputs flow through the dict (not individual `service_user_X` vars) so each call replaces the dict wholesale and inter-call vars-leak can't happen at play-vars scope. Tag the import call with the role tag. Roles with non-standard variants (alternate dir mode, extra groups like `media`/`render`) currently roll their own — extend the helper before duplicating.
- `podman_secret` — `import_role: name: podman_secret, tasks_from: secret` with `podman_secret_args: { name: <n>, data: <d> }` creates or rotates a podman secret. Optional dict key: `user` (default `root`; non-root targets a specific rootless podman). Inputs flow through the dict so each call replaces the dict wholesale and inter-call vars-leak can't happen. Tracks the data sha256 as a label on the secret itself and force-recreates only when content drifts (the upstream module is name-idempotent only and silently ignores rotations). Exposes `<name>_rotated` and `<name>_existed_before` so callers can wire restart triggers and detect post-bootstrap rotations of state that consumes the secret only at first creation (e.g. Django superuser via `SUPERUSER_PASSWORD` — fail loudly and tell the operator to run `manage.py changepassword` rather than rotate silently).
- `systemd_unit` — `import_role: name: systemd_unit, tasks_from: {install,dropin,service}` installs/validates a unit, reloads systemd, and (for `service`) pre-pulls referenced podman images and starts/restarts the unit. Inputs flow through a single dict `systemd_unit_args` (keys: `src`, `dest`, `user`, `condition`, `start`, `restart`, `reload`, `verify`, `for`). The dict form is load-bearing: with the old individual-var form (`systemd_unit_user: x`), `import_role: vars:` leaks values as play-scoped vars and a previous caller's `user: github_runner` would bleed into the next caller that didn't explicitly override. The dict gets replaced wholesale per call (it's a single name, not a merge), so each receiver sees exactly the keys its caller passed and nothing else. `install` dispatches to copy or template based on `src`'s extension (`foo.service.j2` → template under the calling role's `templates/`, `foo.service` → copy under `files/`); the role strips `.j2` for the on-disk dest. `dropin` writes `<scope>/<unit>.d/override.conf` for a unit shipped elsewhere; pass `for: <unit>` and the same extension-tagged `src`. Drive restarts via `restart: "{{ <unit-dest>_result.changed or <other>_rotated }}"` (e.g. `healthchecks_service_result.changed`) so config / secret changes propagate. The legacy `systemd_unit.changed` register still works for backward compat but collides with the role name and is clobbered if a single play installs two units — when touching an existing caller, migrate it to the per-unit fact. **Do not use Ansible handlers for service restarts** — handlers run at end-of-play, which doesn't compose with the helper's pre-pull-then-start ordering, and the inline OR chain keeps the set of restart triggers explicit at the call site instead of scattered across `notify:` annotations. When you add a new config file or secret, append its `.changed` / `_rotated` to the chain. Always prefer this helper over a hand-written `template:` + `systemd:` pair.
- `macvlan` — list-driven per-host setup of macvlan ifaces and (optionally) their matching podman networks. The role lives in `site.yml`'s role list after `podman` so the podman_network module is usable. host_vars supplies a list `macvlan_blocks`; `main.yml` iterates entries and runs `tasks/install.yml` per block, bringing up one `<name>.service` systemd unit per iface and, when `podman_network: true`, creating a podman macvlan network of the same name on the same parent. Each entry is a dict: `name` (iface + unit basename — use `mac_<vlan>`: `mac_mgmt` for the mgmt VLAN, `mac_iot` for the IoT VLAN, etc.), `parent` (parent iface — `eth0` for mgmt, netplan-named VLAN subif like `iot` for tagged subifs), `block` (host's allocation CIDR within the parent VLAN's subnet — drives the iface route and the podman ip_range), `host_ip` (optional, defaults to `block | ipmath(0)` — slot 0 of the block), `mac_address` (optional, defaults to a stable `random_mac` seeded by `inventory_hostname_<name>` so sibling blocks on the same host get distinct MACs), `podman_network` (optional bool, default false; when true, install.yml inlines a `containers.podman.podman_network` call with subnet/gateway derived from `ansible_facts[parent].ipv4` and ip_range = upper /29 of `block` — lower /29 = host iface + libvirt VMs, upper /29 = podman containers). Consumers (homeassistant, future per-VLAN services) attach via `--network={{ name }}` and don't need to know the parent / subnet / ip_range plumbing — see `roles/homeassistant` for an example (declares `mac_iot` in its `homeassistant_networks` list and relies on host_vars to provision the network). Treat the podman ip_range as a flat netavark allocation pool — every IP in the cidr is a usable container slot, no synthetic network/broadcast/gateway inside it (the real gateway is on the parent VLAN).

### healthchecks is not dead code
Grepping `roles/` for `healthchecks.lab.fahm.fr` / `hc-ping` / similar will turn up zero callers, but the service is live: it receives heartbeats from sources outside this repo (cron jobs on personal machines, etc.). A repo-only grep is not sufficient evidence to retire the role.

### Kuma monitor list is UI-managed
The kuma sqlite DB ([/mnt/services/sqlite/kuma/](roles/kuma/)) carries monitor definitions that are added/edited via the Kuma UI, not in this repo. Don't migrate to IaC ("dump monitors to vaulted YAML + restore on bootstrap") — the monitor set extends beyond this repo's scope and overwriting on bootstrap would clobber UI-side state. The kuma.service systemd unit going down is covered by `systemd_service_unit_failed_state`; a Kuma-up-but-empty-DB silent failure is a real but rare gap that's not worth specifically alarming on today.

## Testing Guidelines
The harness lives in `test/` (Python, asyncio-based; the previous `*.sh` shims are gone).

- `test/testrole.py <role>` boots a `box` QEMU VM by default and applies the role end-to-end. Pass `--machine {minimal,box,lab,pug}` to pick a different variant, `--ubuntu noble` to target a different release codename, `--keep` to leave the machine running for SSH debugging, and `--timeout SECONDS` to bound the run (default 30 min). Push CI only fans out to `box` and (when in `ci-minimal-roles.txt`) `minimal`; `lab` and `pug` are reserved for on-demand debug + the nightly full-universe run. The default flow runs check-mode, applies the role, runs it a second time to verify idempotence, then runs `_verify.yml` if present. Disable phases with `--no-checkmode` or `--no-idempotence` for tight dev loops; `--benchmark` adds harness phase timings + ansible's `profile_tasks` callback when investigating slowness. Output is colorized and written to `test/out/<machine>.<role>.ansi`; on failure the systemd journal is collected and the last 50 lines are tailed to stdout.
- `test/testall.py` fans out role × machine combinations across N concurrent workers (`--jobs N`, default 5). It accepts the same `--ubuntu`, `--checkmode`, `--idempotence` flags and forwards them. `--retry-failed` rereads `test/out.tsv` and reruns just the failing rows; `--include-role X,Y` restricts to the named roles; `--exclude-role X,Y` skips them (composes with `--include-role` as subset-then-exclude). All three flags merge into the existing `out.tsv` so untouched triples survive; an unfiltered run replaces it. The joblog is tab-separated with the previous run preserved at `test/out.tsv.prev`.
- Exit codes are meaningful: `0` success, `1` converge failure, `124` per-test timeout, `125` idempotence failure, `130` user-cancelled. The joblog records the integer so you can sort/filter by failure mode.
- Keep the `box` / `pug` / `lab` artifact trees under `/mnt/scratch/qemu/<codename>/` (Linux) or `packer/artifacts/<codename>/` (Mac) for the QEMU profiles. Rebuild any of them with the Packer commands above. The `minimal` variant lazily fetches the upstream Ubuntu cloud image into `<imagedir>/cloud-images/`; macOS hosts also need `xorriso` (`brew install xorriso`) for the cloud-init seed iso when `cloud-localds` isn't present. Profile shapes: `minimal` (vanilla cloud image, arch-portable, ext4); `box` (minimal ZFS-on-root CI fixture: single-disk rpool, no extra pools); `pug` (matches pug prod: single rpool + apoc); `lab` (matches lab prod: mdadm-EFI + 3-disk mirror rpool + rpool/swap zvol + dozer/tank/mouse). All variants run on arm Mac via HVF — see "Test Environment Design" below for the direct-boot path that makes ZFS variants work without a ZBM aarch64 prebuilt.
- Include `ansible-playbook ... --check` and Terraform `plan` snippets in reviews so idempotence and drift are obvious.

### Debugging prod hosts directly
SSH to `lab` / `pug` / `bunk` works out of the box (see [hosts.ini](hosts.ini); `ssh lab 'sudo …'` works without further setup, key + sudoers already configured). When investigating a production issue — service logs, daemon state, hardware probes, on-disk config — connect and run the diagnostics yourself rather than asking the user to paste output. Examples: `ssh lab 'sudo fan2go detect'`, `ssh lab 'sudo journalctl -u nginx -b --no-pager | tail -200'`, `ssh pug 'systemctl list-units --failed'`. Pasting the user's command output back in chat round-trips every iteration through them; self-service triage is faster. Caveats: don't run state-mutating commands without asking (restart services, edit configs, etc.) — diagnostics only.

### Test Environment Design

The harness has four variants — `minimal`, `box`, `lab`, `pug` — chosen by deliberately collapsing axes that don't vary in this stack:

- **Bootloader / EFI**: ZFSBootMenu via rEFInd, EFI-booted, on every prod host. `chroot.sh` installs `linux-generic` with `--no-install-recommends` to skip grub2 and registers ZBM via `efibootmgr`. No GRUB anywhere — don't add it as a test variant.
- **Filesystem**: ZFS-on-root via debootstrap for the prod-shaped variants. `minimal` is ext4 only because Ubuntu cloud images ship that way — it's a "stranger baseline" that catches role assumptions about ZFS / vagrant / packer-baked state, not a config under test. No prod host runs ext4.
- **Architecture**: x86_64 on Linux dev hosts (KVM); arm64 on Mac (HVF). All variants work on both. ZBM has no official aarch64 prebuilts (v3.0.1 is x86_64-only; v3.1.0+ supports aarch64 only via build-from-source) and we don't run a ZBM build pipeline, so on aarch64 the ZFS-rooted variants (`box`/`lab`/`pug`) bypass the firmware boot chain entirely: the packer build ships the on-pool kernel + initrd next to the qcow2 (provision.sh extracts them right before pool export; packer's file provisioner downloads them into the artifacts dir), and qemu boots them directly via `-kernel`/`-initrd`. See `notes/zbm-aarch64-kexec-bug-report.md` for the EDK2-on-aarch64 kexec panic that motivated the workaround.
- **Disk topology**: the only axis that genuinely varies between VM variants. All pools are baked into the packer images by `packer/scripts/pools.sh` (apoc/dozer/tank/mouse, depending on variant), so there's no post-boot disk-setup step. `box` ships 1 disk (single rpool, no extra pools), `pug` ships 3 disks (1 rpool + 2 apoc), `lab` ships 9 (3 mirror rpool + 6 for dozer/tank/mouse). Editing pool topology requires a packer rebuild (15-30 min) — the previous post-boot `test/disks/<variant>.sh` mechanism is gone.

Push CI fans out only to `box` (and `minimal` for roles in `ci-minimal-roles.txt`). `lab` and `pug` images exist for two reasons: to exercise the `chroot.sh` / `provision.sh` / `pools.sh` scripts against the real prod-host shapes on every `packer-build`, and to provide an on-demand harness target for `testrole.py --machine lab/pug` when role behaviour needs validating against the actual prod host_vars (notably for producer roles like `data` / `media` / `scratch` / `minio` against the real multi-pool layout — box exercises them too, but with `zfs_dozer_filesystem` / `zfs_tank_filesystem` repointed at rpool so the datasets land flat under rpool/). Nightly runs the full universe across all variants.

Three packer images, all first-class regression targets:

- `pug` — single-disk rpool + apoc mirror. Matches the pug prod host shape.
- `lab` — mdadm-EFI + 3-disk mirror rpool + rpool/swap zvol + dozer + tank (raidz2) + mouse. Matches the lab prod host shape. Doubles as the multi-disk regression for `chroot.sh` / `provision.sh` and as a copy-paste reference for provisioning new lab-class hosts.
- `box` — single-disk rpool, no extra pools. Minimal ZFS-on-root CI fixture; producer roles run here too but their dozer/tank-keyed dataset paths land under rpool/ via `host_vars/box.yml`'s `zfs_{dozer,tank}_filesystem: rpool`.

**Runner containerisation + DooD.** The `github_runner@.service` template (system-scope, runs as root) starts one podman container per registered instance.

*Ephemeral + JIT lifecycle.* Each container start mints a single-use JIT config by POSTing to GitHub's `/actions/runners/generate-jitconfig` with a host-level fine-grained PAT (vaulted in `host_vars/<host>.yml` as `github_runner_pat`; scope = `Administration: Read and write` on each target repo, Metadata: Read auto, NOT `Actions: Read/write` — overscoped). The runner registers, picks up one job, runs it, deregisters itself, exits. systemd's `Restart=always` brings up the next container, which mints its own JIT blob. No persistent runner state on disk -- no `.runner`/`.credentials` carryover between starts.

*Container userns.* Standard CLAUDE.md fake-root pattern: `--user 0:0` + `--uidmap=0:0:65536 --uidmap=+0:{{ github_runner_user.uid }}:1` (matching gidmap). podman runs as real root in the unit; in-container uid 0 demotes to host `github_runner` via the +0 override. (Two earlier branch attempts kept podman itself rootless for defense in depth — system-scope-with-`User=github_runner`, then user-scope under github_runner's user manager — both hit failures, see the unit template's [Unit] rationale comment for the history.)

*Path-mirroring.* Two bind mounts have matching source/dest paths on host and inside container, which is what makes `container:` workflows work over DooD:
- `--volume /opt/actions-runner:/opt/actions-runner:rw`: the runner binaries themselves. Installed on host by [roles/github_runner](roles/github_runner/tasks/main.yml) from the tarball pinned in [roles/github_runner/vars/main.yml](roles/github_runner/vars/main.yml) -- a runner version bump is an ansible-only change (no runner-image rebuild needed). The runner image itself ships no actions-runner binaries.
- `--volume /mnt/scratch/github_runner/workdir_%i:/mnt/scratch/github_runner/workdir_%i:rw`: the runner's `_work` tree (per-instance). `--env RUNNER_WORK_FOLDER=/mnt/scratch/github_runner/workdir_%i` points actions/runner at it via the JIT config's `work_folder`.

When a `container:` workflow runs, actions/runner spawns `docker create -v /opt/actions-runner/externals:/__e:ro -v /mnt/scratch/.../workdir_%i/_temp:/__w/_temp ...`, and docker (resolving sources on the host via the bind-mounted DooD socket) finds those paths because they exist at the same absolute path on both sides.

*DooD.* The runner container bind-mounts github_runner's rootless podman socket at `/var/run/docker.sock`. `--env CONTAINER_HOST=unix:///var/run/docker.sock` + `--env DOCKER_HOST=unix:///var/run/docker.sock` route in-container `podman`/`docker` invocations to the socket, so builds happen on the host's rootless podman (which has CAP_SYS_ADMIN in its userns for mount setup -- nested rootless inside the runner container does not, and would fail "remount /, flags: 0x44000: permission denied" during ApplyLayer).

*Host-level PAT secret.* `github_runner_pat` podman secret (mount-type, so the value never lands in `/proc/<pid>/environ` visible via DooD `docker exec`) is shared across every runner instance on the host -- the generate-jitconfig API call is per-repo, not per-runner. One secret, many instances. Operator-managed: mint as a fine-grained PAT in the GitHub UI, vault into `host_vars/<host>.yml`. Scaling up = bump the `github_runner_instances` count and re-run the role.

*Other notable args.* `--cgroups=split` for per-unit resource accounting (`Delegate=yes` makes the subtree usable); `--sdnotify=healthy` gates the unit's `active` state on the healthcheck; `--health-cmd "pgrep -f Runner.Listener"` (procps is in the image for this); `--security-opt seccomp=unconfined` (jammy podman 4.9.3's bundled seccomp profile predates kernel 6.6's `fchmodat2` syscall and returns EPERM for it, which breaks GNU tar 1.35+ inside; threat model unchanged because the +0 uidmap already caps in-container compromise at unprivileged `github_runner`); `--pull=newer` so a fresh `runner-image` build propagates on the next container start; `--stop-timeout 60` to fit actions/runner's ~50s GitHub long-poll without SIGKILL noise on every restart cascade.

Workflows that declare `container:` blocks get their sibling containers spawned on the host's rootless podman via the bound socket -- the runner container itself doesn't claim `/dev/kvm`; the qemu-running siblings do.

Sibling workflow containers use `--device /dev/kvm --group-add 1` for KVM acceleration. The `1` is the **container-side gid** that the rootless userns maps the host's kvm gid to — not the host kvm gid itself. Three pieces have to line up:
1. `service_user_extra_groups: [kvm]` on the github_runner user (so the host user can read `/dev/kvm` at all).
2. `subid_extra_subgid_groups: [{ user: github_runner, group: kvm }]` in [host_vars/lab.yml](host_vars/lab.yml) so `/etc/subgid` carries `github_runner:<kvm-gid>:1` — making the kvm gid delegatable into a userns.
3. After changing `/etc/subgid`, run once on lab: stop all `github_runner@*` instances, `sudo -u github_runner podman system migrate`, then start them again — podman caches the gidmap and won't re-read the new line otherwise.

Why the literal `1`: rootless podman builds the container gidmap from `/etc/subgid` shortest-range-first. With our two-range layout (one 65k range + one single-gid range for kvm), the single-gid range lands at container gid 1 — so host kvm = container 1. Why not `--group-add keep-groups`? It's a podman-CLI extension; the runner talks to podman over the docker-compat socket, where Docker semantics interpret `keep-groups` as a literal group name and the container fails with "no matching entries in group file". Why not `--group-add <kvm-host-gid>` directly? That host gid is unmapped inside the container's userns (or maps to a high subuid), so the supplementary group doesn't grant access to a host-gid-`<kvm>` file. The literal `1` is brittle to one thing only: adding a *second* single-id range to `subid_extra_subgid_groups` for github_runner could shift ordering — re-verify if you ever do that.

The `gitea_runner` role still ships a stripped-down Gitea Actions runner on lab (labels: `gitea/runner-images:ubuntu-*`, no KVM, no `/mnt/scratch/qemu` mount, no custom image). It's there for ad-hoc Gitea repo workflows; this repo's CI does not run through it.

## Commit & Pull Request Guidelines
History favors short, imperative subjects such as “Fix dnscrypt” or “Add profilarr”; prefix with a role when it helps clarity (`wireguard: rotate peers`). Each PR should summarize the motivation, list impacted hosts or roles, and link related issues. **Splitting a multi-change worktree into per-finding commits.** When several findings have been edited into the worktree together (e.g. during a multi-item `/improve` or `/review` pass), commit each one separately by driving `git add --patch=<file>` non-interactively: `printf 'y\nn\nn\n' | git add -p <file>` selects hunks in file order (y = stage, n = skip). Hunks within ~3 unchanged lines of each other are auto-merged into one — answer `s` first to split before answering y/n if you need finer granularity. Verify each commit's staged content with `git diff --staged` before `git commit`. Do **not** revert the worktree to redo each edit from scratch as a way to get clean commits: it wastes effort, loses the just-tested state, and risks dropping a comment/decision detail the second time around.

### Vault ids: `prod` vs `test`
Two passwords, two scopes. Configured in [ansible.cfg](ansible.cfg) as `vault_identity_list = prod@vault-client.sh, test@vault-client.sh` with `vault_id_match = True`.

- `prod` — encrypts `group_vars/prod.yml` and prod host_vars (lab, pug, bunk). Stays on local workstations only; never lands in CI.
- `test` — encrypts `group_vars/test.yml` and test host_vars (currently just `host_vars/box.yml`). Available to CI as the Gitea repo secret `HOMELAB_VAULT_PASSWORD_TEST` so the test harness can decrypt test vars.

[vault-client.sh](vault-client.sh) follows ansible's "client" password-script protocol: ansible only passes `--vault-id <id>` to scripts whose filename ends in `-client`, hence the suffix. The script accepts both `--vault-id <id>` and a positional `$1` for ad-hoc invocations. Lookup order per id: env var `HOMELAB_VAULT_PASSWORD_<UPPER_ID>` first (CI), then macOS keychain entry `homelab-vault-<id>`, then Linux file `~/.config/homelab/vault-pass-<id>` (mode 0400). To add a new password locally: `security add-generic-password -a ak -j "vault password ansible (<id>)" -s homelab-vault-<id> -w` on Mac; `install -m 0400 /dev/stdin ~/.config/homelab/vault-pass-<id> <<<"$pw"` on Linux.

When encrypting a new value with `ansible-vault encrypt_string`, pass `--encrypt-vault-id prod` (or `test`) so the resulting envelope carries the right label — otherwise `vault_id_match = True` will refuse to decrypt it.

Most other `*.service.j2` templates almost certainly carry the same numbers because they were copied from a common ancestor. Visit each, calibrate to the service's real cold-start time and graceful-shutdown behaviour, and tighten where the original numbers are pure padding.
- **Standardize container timezones.** Service units are inconsistent about how (and whether) they set TZ — some use `--tz=`, some pass `TZ=` env, some leave it unset and inherit UTC. Pick one mechanism (`TZ=` env is more portable; `--tz=` is podman-only but covers programs that ignore `TZ`) and a policy: local time for user-facing/log-read/cron-driven services, UTC for databases and metrics stacks. Apply across `roles/*/templates/*.service.j2` as roles get touched.
- **Audit cheap systemd hardening across service units.** Most `*.service.j2` units stop at `NoNewPrivileges`/`ProtectSystem=strict`/`ProtectHome` and skip the zero-cost directives that apply to most network daemons: `RestrictAddressFamilies=AF_UNIX AF_INET AF_INET6`, `RestrictNamespaces=yes`, `RestrictRealtime=yes`, `RestrictSUIDSGID=yes`, `LockPersonality=yes`, `MemoryDenyWriteExecute=yes` (where the runtime doesn't JIT — CPython is fine, Node.js is not), `ProtectKernelTunables`/`ProtectKernelModules`/`ProtectKernelLogs`/`ProtectControlGroups`/`ProtectClock`/`ProtectHostname`, `ProcSubset=pid`, `SystemCallArchitectures=native`, `SystemCallFilter=@system-service`, `CapabilityBoundingSet=` empty. Sweep `roles/*/templates/*.service.j2` and add the applicable subset; podman units (where the container itself is the security boundary) need a different subset than native python/go daemons.
- **Strict rp_filter trap for multi-homed routing.** `roles/firewall/files/sysctl-hardening.conf` pins `net.ipv4.conf.{all,default}.rp_filter = 1` (strict). Today routing is symmetric — WAN, wg0, podman bridges, macvlan all have a single coherent return path. The day someone introduces policy-routed asymmetry (dual-WAN failover, second egress, source-routing) strict rp_filter silently drops replies arriving on a different iface than the kernel would have routed out, and the failure looks like packet loss not a config bug. Loosen to 2 (loose) or 0 (off, firewall does the work) before any policy-routing change.
- **Migrate per-service user handling onto uidmaps.** The intended pattern is `--uidmap=0:0:65536` + `--uidmap=+0:<svc>_user.uid:1` (and the gid pair) — container root drops to the host service uid via the namespace, and files written to bind-mounted dirs land owned by that uid. PUID/PGID env vars (linuxserver images) and `--user <uid>:<gid>` in-container do the same job a different way and **don't compose with the uidmap**: the `+0:N:1` override unmaps in-container uid N, so any `chown N:M` inside the entrypoint returns EINVAL (see healthchecks for the canonical failure). For linuxserver images keep the uidmap and set `PUID=0` / `PGID=0` to opt into their "root mode" (skips the usermod/chown/privilege-drop dance). For other images, prefer `--user root` + uidmap and drop any `--user uid:gid`. Audit each `*.service.j2` and migrate.
- **Audit `--init` across podman service roles.** [roles/compta/templates/compta.service.j2](roles/compta/templates/compta.service.j2) sets `--init` so podman's `catatonit` runs as PID 1 inside the container, reaping zombies and forwarding SIGTERM to the real entrypoint. Most service images aren't process-supervising at PID 1 (ruby/python/node/distroless app binaries are common) and benefit from the same. Sweep `roles/*/templates/*.service.j2` and add `--init` where the image's entrypoint isn't itself a supervisor (tini, dumb-init, s6-overlay, etc.).
- **Audit `backup: true` on secret-bearing templates.** Repo policy (per agent memory) is "every config-writing task gets `backup: true` for on-disk traceability, even trivial files." That's wrong for files containing vaulted secrets — every rotation leaves a stale `<file>.<ts>~` containing the *previous* secret next to the live one, accumulates indefinitely, and ships across `zfs send/recv` to replication targets. The vault file is the canonical history. Sweep `roles/*/tasks/*.yml` for `template:` tasks whose destination renders any `*_password` / `*_token` / `*_secret` Jinja var, and either drop `backup: true` or hook a per-file cleanup that prunes `<file>.<ts>~` older than N days. [roles/getmail/templates/getmailrc.j2](roles/getmail/templates/getmailrc.j2) (Gmail app password) is the canonical example flagged by review.
- **Unify `ansible_devices` filter chains across hd_idle / hdparm / smart.** Three roles loop over `ansible_devices` with three different filter spellings (`hd_idle` excludes md/dm/loop/sr/ram/zd by name prefix and filters rotational=1 + removable=0 + links.ids non-empty; `hdparm` filters rotational only; `smart` filters nothing and matches by `nvme-`/`ata-` prefix later). There's no clean ansible-fact discriminator for "real physical block device" — `ansible_devices[x].virtual` looks like the right thing but on modern kernels (`/sys/block/<x>` is always a symlink) ansible's logic at `module_utils/facts/hardware/linux.py:733-748` sets `virtual=1` for every disk, so it's effectively unusable. The working approach is the name-prefix blacklist hd_idle uses. When next touching hdparm or smart, adopt hd_idle's shape — and consider hoisting a shared `host_physical_disks` lazy dict into `group_vars/all.yml` once two consumers agree on the same return shape
- **github_runner: tamper-detection on host-side actions/runner.** The unit bind-mounts host `/opt/actions-runner` `:rw` into the runner container so DooD path-mirroring works for `container:` workflows. `:rw` lets a malicious workflow persist tampering to the host (e.g. rewrite `run.sh` or `bin/Runner.Listener`). The cleaner `:O` (overlay) flag would discard in-container writes on exit but is fundamentally incompatible with the `+0 uidmap` demotion — `:O`'s bind-mount source lives under `/var/lib/containers/storage/overlay-containers/<cid>/userdata/` which podman creates 0700 root-only, and crun's source-path lookup runs as in-userns root (= host `github_runner`), hitting EACCES on traversal. Two paths to cover the gap without overlay: (a) extend `github_runner_image_rotate.sh` (or a sibling 5-min timer) to hash-check the actions-runner tree against the `sha256` pinned in [vars/main.yml](roles/github_runner/vars/main.yml) and re-extract from `.runner.tar.gz` on drift; (b) move `/opt/actions-runner` onto a tmpfs and re-extract from the tarball at host boot. (a) is the cheaper of the two and gives both tamper-detection AND self-healing. Skip the more invasive separate-podman-storage-root workaround unless we hit another `:O`-blocked use case.
